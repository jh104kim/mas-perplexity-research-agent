# 07. Further Work

## 문서 역할

이 문서는 MVP에서 제외한 기능을 관리한다. 여기 있는 기능은 바로 구현하지 않고, MVP가 통과한 뒤 필요성과 비용을 다시 판단한다.

## MVP 이후에 검토할 기능

아래 기능은 당장 구현하지 않는다. MVP가 안정적으로 동작한 뒤에 검토한다.

## 보류 기능 요약

| ID | 기능 | 현재 상태 | 재검토 조건 |
| --- | --- | --- | --- |
| FW-01 | Perplexity API 직접 호출 | 보류 | HITL 방식이 반복 작업에 비효율적일 때 |
| FW-02 | 웹 대시보드 | 보류 | CLI 사용이 불편해질 만큼 실행 이력이 늘어날 때 |
| FW-03 | Obsidian 저장 | 보류 | 파일 산출물 구조가 안정된 뒤 |
| FW-04 | Slack 알림 | 보류 | 자동 실행이나 팀 공유가 필요할 때 |
| FW-05 | 예약 실행 | 보류 | 단일 실행 Workflow가 안정된 뒤 |
| FW-06 | 멀티 사용자 기능 | 보류 | 개인 학습용 범위를 벗어날 때 |
| FW-07 | Perplexity Web UI 자동화 | 보류 | HITL 수동 실행도 부담이 될 때 |

## 1. Perplexity API 직접 호출

Perplexity API를 코드에서 직접 호출하는 방식이다.

보류 이유:

- 현재 목표는 API Key 없이 HITL로 검증 가능한 Workflow를 만드는 것이다.
- API 비용, 인증, 호출 제한, 에러 처리가 추가된다.
- 사용자가 직접 Perplexity 결과를 확인하고 Markdown으로 가져오는 방식이 더 단순하다.

## 2. 웹 대시보드

리서치 실행 이력과 보고서를 웹에서 보는 기능이다.

보류 이유:

- 현재 핵심은 Workflow 검증이다.
- CLI만으로도 MVP 목적을 달성할 수 있다.

## 3. Obsidian 저장

최종 보고서를 Obsidian Vault에 자동 저장하는 기능이다.

보류 이유:

- 파일 저장 구조가 먼저 안정되어야 한다.
- 개인 Vault 경로 정책이 추가로 필요하다.

## 4. Slack 알림

리서치 완료 후 Slack으로 알림을 보내는 기능이다.

보류 이유:

- 외부 연동이 늘어난다.
- MVP에서는 보고서 파일 생성만으로 충분하다.

## 5. 예약 실행

정해진 시간마다 특정 주제를 리서치하는 기능이다.

보류 이유:

- 실패 재시도, 중복 실행, 로그 관리 정책이 필요하다.
- 먼저 단일 실행 Workflow를 안정화해야 한다.

## 6. 멀티 사용자 기능

여러 사용자가 각자 리서치 실행 이력과 보고서를 관리하는 기능이다.

보류 이유:

- 현재 목적은 개인 학습 및 위키 축적이다.
- 사용자 계정, 권한, 저장소 분리 정책이 추가로 필요하다.

## 7. Perplexity Web UI 자동화

브라우저를 직접 조작해서 Perplexity Web UI를 사용하는 방식이다.

보류 이유:

- UI 변경에 취약하다.
- 자동화 실패 원인이 늘어난다.
- MVP 검증에는 HITL 수동 실행 방식이 더 단순하다.

보류 기능을 MVP 범위로 올리려면 아래 문서를 함께 수정한다.

| `05-test-scenarios.md` | Given/When/Then 시나리오 추가 |
| `06-implementation-roadmap.md` | 구현 Phase와 완료 기준 추가 |

## 보류 기능 재검토 체크리스트

보류 기능을 다시 검토할 때는 아래 질문에 답한다.

- 현재 MVP 목표를 이미 통과했는가?
- 이 기능이 개인 학습 및 위키 축적 목적에 꼭 필요한가?
- 기존 CLI Workflow를 복잡하게 만들지 않는가?
- 새 외부 연동 비용이나 보안 위험이 감당 가능한가?
- 테스트 시나리오와 완료 기준을 명확히 쓸 수 있는가?

위 질문에 대부분 "예"라고 답할 수 있을 때만 PRD 승격을 검토한다.
